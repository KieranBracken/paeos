# G5 — Roles + Genesis Protocols

## Forcing clause
PAEOS-4 §5 (roles = stance × domain, mutation matrix) · §7 (protocols per router cell) ·
PAEOS-3 SVK-6/7 (one Builder + four naive protocols; rest demand-paged) · package: **G-ROLES**.

## Writes
- **Four role policies** (`loading: at-spawn`): Builder=(GENERATE,WORK),
  Adversary=(FALSIFY,WORK), Judge=(SELECT,WORK), Classifier=(SELECT,GOALSTRUCT). The other
  cells are **demand-paged** — creating them now violates no-rule-before-demand (PAEOS-3 §4).
  Each body transcribes its PAEOS-4 §5.3 mutation-matrix row verbatim.
- **Four genesis protocols** (`loading: routed`): proto-direct / proto-checked /
  proto-adversarial / proto-escalate, mapping the 9 (v,r) router cells → 4 protocols
  (SC-11). **Naive by construction**: `fanout: 1`, no retry logic, no quality heuristics.
  Refinement is evolution's job, not genesis's — shipping clever protocols would be rules
  without scars.

## Inputs
`roles/`, `protocols/` (G-ROLES output) · PAEOS-4 §5.3 matrix · SC-11 mapping table.

## Exit evidence
Eight policies admitted; each compiles into a valid Context under the compiler (verifiable
once the compiler exists); the SC-11 mapping covers all 9 cells exactly once.

## Why so few, so naive
Demand-paging (PAEOS-3 §2) and no-rule-without-a-scar (K6): the system starts with the
minimum that lets the first goal move, and every later role/protocol/refinement must be
born from an actual incident. A rich genesis policy set would be the cargo-cult failure
mode PAEOS-0 exists to prevent.
