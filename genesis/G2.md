## Why the merge was forced
The original sequence had G1 (GenesisRecord) precede G2 (schemas). But:
- Every object header requires `schema: HASH` (PAEOS-4 §2.2).
- The GenesisRecord is an object, so it needs a schema in the store.
- Schemas enter via WRITE, which validates against store-loaded schemas — none exist yet.

Circular. The resolution (B-03): a hardcoded **meta-schema sentinel**
(`schema: "sha256:meta.v1"`, part of the kernel spec text, embedded identically in every
conformant implementation) validates exactly two shapes — schema-kind Policy artifacts and
the GenesisRecord — admitted together in one commit. After this batch, all further objects
validate against the now-store-resident schemas.

See `G1.md` for the actual batch contents and exit evidence.
